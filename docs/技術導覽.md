# 技術導覽（給技術新手 PM 看的版本）

這份文件目的：讓不寫 code 的 PM 也能理解「My Stock Portfolio」這個 App 是怎麼運作的、用了哪些技術、為什麼選這些技術、還有什麼別的選擇。

> **這版內容是「實際讀過 code」之後寫的**，跟 CLAUDE.md / README 的描述有幾處不同。和文件不一致的部分，整理在 `docs/code-review-notes.md`，建議搭配閱讀。

---

## 1. 這個 App 在做什麼？

一個**個人理財 PWA**（Progressive Web App，可安裝到手機/桌面的網頁 App），主要功能：

- 管理台股 / 美股**持股**，手動更新報價、看漲跌
- 管理**現金帳戶**（含台灣銀行清單）、**支出**、**預算**
- 看 24h / 7d / 30d 的**現值走勢**
- 登入（**Google 或 Email/Password 二選一**）後，把資料同步到雲端，多裝置共用

技術上的關鍵特色：
- **單頁應用 (SPA)**：一個 HTML 檔載入後，所有切換都在前端完成
- **零自架後端**：除了 Firebase（Auth + Firestore）之外沒有自架 server，連報價都直接從前端打 API
- **雲端 + 本地雙寫**（注意：**不是真正的「離線優先」**，見第 4 節說明）

## 2. 整體運作原理（一張圖看懂）

```
[使用者瀏覽器]
   │
   │ 1. 開啟網頁 → 從 GitHub Pages 下載靜態檔（HTML/JS/CSS）
   │
   ├──→ 2. 讀寫資料：window.localStorage（瀏覽器內建的小型儲存空間）
   │
   ├──→ 3. 抓報價：依市場分流
   │     ├─ 美股：Finnhub
   │     └─ 台股：TWSE → TPEX（多層 fallback）
   │
   ├──→ 4. 抓匯率：open.er-api（USD/TWD）
   │
   ├──→ 5. 抓銀行清單：data.gov.tw（建現金帳戶用）
   │
   └──→ 6. 登入後：所有寫入都先送 Firebase Firestore，再 round-trip 回本地
                                 ↑
                       其他裝置登入 → 從這裡 realtime listener 推下來
```

**關鍵流程：使用者新增一筆持股**（實際 code 行為）
1. UI 表單 → `portfolioService.upsertHolding(...)` → 先 `ensureCloudWritable()` 檢查（**未登入 / 離線 / Firestore 沒就緒會擋掉**）
2. `mirrorToCloud(...)` → **先 `setDoc` 寫到雲端 Firestore**
3. 雲端寫完 → `applyCollectionRecordLocally(...)` → 把雲端的「正規版」資料寫回本地 localStorage
4. 同時 Firestore realtime listener 也會把這筆收回（其他裝置會自動同步）

> **重點 trade-off**：這是「**雲端先、本地後**」的同步模式，不是真 local-first（見第 4 節）。

## 3. 前端：使用者看到的部分

### React 19
**做什麼**：構建 UI 的 JavaScript library。把畫面切成一個個「元件 (component)」，資料變了就自動重新渲染。

**為什麼選它**：生態系最大、找人最容易、社群資源多。

**替代選擇**：
- **Vue**：學習曲線比較平緩，台灣有些團隊偏好
- **Svelte**：編譯時就把模板轉成 vanilla JS，bundle 更小、執行更快
- **Solid**：API 像 React 但效能更好（沒有 Virtual DOM）
- **Angular**：大型企業用得多，框架更「重」、約束更強

### Vite 7
**做什麼**：開發時的本地 server + 打包工具（把 JS/CSS/圖片打包成瀏覽器看得懂的 bundle）。

**為什麼選它**：開發時改 code 後**毫秒級熱更新 (HMR)**，比舊的 Webpack 快很多。

**替代選擇**：
- **Webpack**：老牌，配置複雜但彈性大
- **Turbopack** / **Rspack**：用 Rust 寫的新世代 bundler，更快
- **Parcel**：零設定，適合小專案

### Ant Design 6（自訂主題色 #44A194）
**做什麼**：UI 元件庫，提供按鈕、表單、表格、Drawer、Modal 等現成元件。`src/main.jsx:23-30` 把主色覆蓋成偏綠的 `#44A194`，圓角 10px、字體優先 PingFang TC / Noto Sans TC。

**為什麼選它**：元件最齊全、中文世界使用者多、表格與表單在後台類 App 很強。

**替代選擇**：Material UI、Chakra UI、shadcn/ui、Tailwind + Headless UI。

### Recharts 3
**做什麼**：畫圖表。這個 App 用 `LineChart`、`BarChart`、`PieChart` 三種，分別畫**現值走勢**、**支出長條圖**、**資產分配圓餅圖**。

**替代選擇**：Chart.js、ECharts、Victory、D3.js。

### `@dnd-kit`（拖曳排序）
**做什麼**：讓持股列表可以拖曳重排。比舊的 `react-dnd` 輕量、accessibility 比較好。

### `animejs`（數字 tweening）
**做什麼**：當總資產數字變動時，做平滑的「滾動動畫」，看起來比直接跳過去更順眼。

### PWA（Progressive Web App）
**做什麼**：透過 `vite-plugin-pwa` 自動產生 service worker 與 manifest。讓網頁可以**安裝到桌面/手機 home screen**、**可離線開啟**（讀資料）、看起來像原生 App。

**這個 App 的 PWA 細節**：
- precache 上限調到 3 MB（目前 bundle ~2.2 MB）
- TWSE / Alpha Vantage 的 API 有 `runtimeCaching` 規則（NetworkFirst，8 秒 timeout，快取 24 小時）
- dev mode 也啟用 PWA（`devOptions: { enabled: true }`）方便測試

**替代選擇**：
- **純網頁**：放棄離線與安裝
- **React Native / Flutter**：跨平台原生 App，效能更好但要過商店審查
- **Electron / Tauri**：把網頁包成桌面 App
- **Capacitor**：把 PWA 包成可上架的 iOS/Android App

## 4. 資料儲存：使用者的資料放在哪裡？

### 本地：`window.localStorage`
**做什麼**：瀏覽器內建的 key-value 儲存空間，每個網域有約 **5–10 MB** 上限。資料以 JSON 字串存著。

**這個 App 怎麼用**：在 `src/db/database.js` 手刻了一個小型「假資料庫」。`PersistedInMemoryTable` 把每張表（holdings、expense_entries、budgets…等 10 張）各自序列化成 JSON 存到 localStorage。query API（`db.holdings.where('[symbol+market]').equals([s, m]).first()`）長得像 Dexie 但底層完全是 localStorage + JSON，**沒有真正的 transaction、沒有索引加速**。

**`db.transaction(...)` 是假的**（`database.js:416-422`）— 它就是直接呼叫 callback，不提供 atomic 保證。

**Persistent tables：**
- `holdings`、`price_snapshots` — 持股 + 走勢圖快照
- `cash_accounts`、`cash_balance_snapshots` — 現金帳戶 + 餘額快照
- `expense_entries`、`expense_categories`、`budgets` — 支出 / 分類 / 預算
- `fx_rates`、`sync_meta`、`app_config` — 匯率、同步狀態、App 設定

**In-memory only：**
- `outbox` — **目前是 dead code**，宣告了但沒人寫入（見 `code-review-notes.md` A4）

**為什麼這樣選**：個人 App、資料量不大、簡單好除錯（DevTools 直接看得到）。

**限制**：5–10 MB 上限、同步 API（讀寫卡主執行緒）、沒 transaction、沒索引。

**替代選擇**：
- **IndexedDB（搭配 Dexie）**：瀏覽器內建的真資料庫，上限數百 MB ~ GB，有索引、async、transaction
- **SQLite (via WASM)**：把 SQLite 編進前端，可寫真 SQL
- **OPFS**：較新的瀏覽器 API，可直接讀寫檔案

### 雲端：Firebase Firestore
**做什麼**：Google 的 NoSQL 雲端資料庫。資料結構是 collection → document。這個 App 把每個使用者的資料放在 `users/{uid}/...` 底下。

**realtime listener**：`onSnapshot(...)` 訂閱 10 個 collection，雲端有變動就自動推下來。

**Firestore Rules（`firestore.rules`）**：每個 user 只能讀寫自己 uid 底下的資料：
```
match /users/{userId}/{document=**} {
  allow read, write: if request.auth.uid == userId;
}
```

**替代選擇**：Supabase（Postgres）、AWS Amplify、PocketBase、自架 Postgres + REST/GraphQL。

### ⚠️ 真相：這不是真正的 Local-first

文件裡寫「local-first 先寫本地再同步雲端」，**但 code 不是這樣做**。

**實際流程**（`portfolioService.js:111-118`）：
```js
const mirrorToCloud = async (collectionName, record) => {
  await writeCollectionRecord(...);          // (1) 雲端先寫
  await applyCollectionRecordLocally(...);   // (2) 再寫本地
};
```

**而且每個 mutation 都會先呼叫 `ensureCloudWritable()`**，這個檢查會在以下情況**直接 throw、阻止寫入**：
- 未登入
- 離線（`navigator.onLine === false`）
- Firestore listeners 還沒就緒
- Firebase 不可用

→ **真正的行為**：
- 「Firebase 已設定 + 有登入 + 在線」：可寫，且為 cloud-first
- 「Firebase 已設定但離線」：**只能讀，不能寫**
- 「Firebase 完全沒設定」（env 變數空著）：才會走純本地模式

> 這對 PM 的意義：不要以「PWA 在地鐵裡也能完整使用」當賣點。離線時只能瀏覽，要新增/修改持股得等回到網路。

**衝突處理**：用 `isRemoteNewer` 比 `updatedAt` 字串做 **last-write-wins**（後寫的覆蓋先寫的）。簡單、適合個人 App，不適合真正多人協作。

**替代選擇（更進階的同步）**：
- **CRDT**（Yjs、Automerge）：真正的多人協作衝突解法
- **真 local-first（先本地、背景送雲端 + outbox）**：例如 Replicache、ElectricSQL、PowerSync

### ⚠️ 登入時會清空本地

`startRealtimeSync(uid)` 一被呼叫（登入流程觸發），會：
1. **先 `clearLocalCloudBackedData()` 把本地所有 cloud-backed table 清掉**
2. 再 subscribe Firestore，把雲端資料拉下來覆蓋

意義：如果你在「未登入」狀態下手動建立一些持股 → 然後登入 → **本地資料會被瞬間取代成雲端版本**（雲端如果是空的，你看到的就是空白）。

## 5. 登入：Firebase Authentication

**做什麼**：處理使用者登入、token 管理、session 保持。

**支援兩種登入方式**：
1. **Google 登入**：`signInWithPopup` 為主，popup 被擋掉時 fallback 成 `signInWithRedirect`（`authService.js:34-45`）
2. **Email + Password 登入**：`signInWithEmailAndPassword`，附上中文化錯誤訊息（`authService.js:65-72`）

**App Check（reCAPTCHA v3）**：在 `firebaseApp.js:32-41`，**只有 production build 且設定了 `VITE_FIREBASE_RECAPTCHA_SITE_KEY` 才會啟用**。
> ⚠️ 但 deploy.yml 目前沒有把這個 secret 注入 build，所以實際上 production 從來沒啟用 App Check。詳見 `code-review-notes.md` A7。

**替代選擇**：Auth0、Clerk、Supabase Auth、NextAuth、自架 OAuth。

## 6. 股價來源：Price Providers

### 為什麼需要「Fallback Chain」（連續備援）

> 想像你要訂高鐵票，第一個 App 壞了就用第二個，第二個壞了就打電話訂——一樣的概念。

每個股價來源都有缺點：
- **Finnhub**：免費版每分鐘 60 次、台股代碼支援差
- **TWSE 官方 API**：CORS 限制、有時下午會掛
- **TPEX 官方 API**：CORS 比 TWSE 更嚴

### 實際的 fallback 順序（讀 code 後修正）

**美股**（`finnhubProvider.js:85-107`）：
- Finnhub `/quote` 取價格 → `/stock/profile2` 取公司名
- **沒有 fallback**，Finnhub 失敗就失敗

**台股**（`finnhubProvider.js:55-83`）— **Finnhub 不在 TW 流程裡**：
1. **TWSE RWD**（`twseRwdProvider.js`）：打 `STOCK_DAY` 查最近 3 個月的日成交，從最新的非空筆找出收盤價
2. **TWSE 全量 snapshot**（`twseProvider.js`）：打 `STOCK_DAY_ALL` 一次拉全部上市股票，本地搜尋
3. **TPEX provider**（`tpexProvider.js`）：再分 5 層 fallback（見下節）

> ⚠️ CLAUDE.md / README / 舊版導覽都把 Finnhub 寫在 TW chain 第一順位，但**實際 code 並沒有**。詳見 `code-review-notes.md` A1。

### TPEX 的 5 層 fallback（最多打 10 次）

`tpexProvider.js:23-31` — 依序嘗試這些 URL，每個 URL 失敗會 retry 1 次（共最多 2 次），8 秒 timeout：

1. **同源 snapshot**：`public/data/tpex_off_market.json`（GitHub Actions 每天更新）
2. **TPEX 官方 API**：`https://www.tpex.org.tw/openapi/v1/tpex_off_market`
3. **使用者自訂 proxy**：若有設 `VITE_TPEX_PROXY_URL`
4. **預設 proxy 1**：`api.codetabs.com`（hardcoded）
5. **預設 proxy 2**：`corsproxy.io`（hardcoded）

> ⚠️ (4)(5) 是**寫死在 code 裡的第三方 proxy**，所有使用者都會經過。文件沒提，需要意識到。

### 同源 TPEX 快照 — 巧妙的解法

TPEX 官方 API 有 CORS 問題，瀏覽器直接打會被擋。解法：

1. **GitHub Actions** 平日 10:10 UTC 自動跑 `update-tpex-snapshot.yml`
2. server 端 `curl` TPEX 官方 API → 存成 `public/data/tpex_off_market.json`
3. 有變動就 commit 進 repo → GitHub Pages 自動 deploy
4. 前端用「同源請求」拿這個 JSON，**繞過 CORS**

### FX 匯率
**做什麼**：USD/TWD。直接打免費的 `open.er-api.com`，不需 API key（`fxProvider.js`）。

### Alpha Vantage 是 dead code
`src/services/priceProviders/alphaVantageProvider.js` 整個檔案沒有 import 它的地方。可以移除（見 `code-review-notes.md` A5）。

### 銀行清單：`twBankDirectoryProvider.js`
建立現金帳戶時的銀行下拉選單。從 `data.gov.tw` 抓 FISC + 分行兩個 dataset，本地 cache 7 天，全部失敗時 fallback 到 hardcoded 33 家主要銀行清單。

**替代選擇（報價）**：
- 自架 proxy server（Cloudflare Workers）
- 付費資料源（Yahoo Finance API、Polygon.io、Twelve Data）
- 券商 API（永豐 Shioaji、富邦 Fugle）

## 7. UI 程式碼結構

### `src/App.jsx` 是個 6,428 行的**巨型元件**

幾乎所有功能（持股、現金、支出、預算、走勢、各種 drawer/modal）都塞在一支檔案。**內部巢狀了至少 8 個 `<Tabs>`** 元件（`activeAllocationTab`、`activeHoldingTab`、`activeCashHolderTab`、`activeExpenseCategoryTab`、`activeBudgetTab`…）。

只有 4 個元件被抽出來：
- `HoldingForm` — 新增/編輯持股
- `CashAccountForm` — 新增/編輯現金帳戶
- `MobileFormSheetLayout` — 手機版底部 sheet 佈局
- `TrendChart` — 走勢折線圖

**`src/services/portfolioService.js` 也很大**（3,437 行）— UI 唯一呼叫的進入點，包含所有 mutation function、normalize 邏輯、constants。

**為什麼這樣**：個人專案、不需分工、跨元件共享 state 不用 threading context。

**正規做法**：拆 page-level component、用 Zustand / Jotai 管 state、用 React Router 處理 tab。

## 8. 部署：GitHub Pages + GitHub Actions

### 流程
1. 開發者 `git push` 到 `main`
2. GitHub Actions（`.github/workflows/deploy.yml`）自動觸發
3. 在 GitHub 雲端跑 `npm ci` + `npm run build`（Node.js 22）
4. 把 `dist/` 上傳到 GitHub Pages

### 環境變數從哪來
所有 `VITE_*` 開頭的變數要在 **GitHub Repository Secrets** 設定。Build 時注入到 bundle 裡。

**目前 deploy.yml 注入的 5 個 secret：**
- `VITE_FINNHUB_API_KEY`
- `VITE_FIREBASE_API_KEY`
- `VITE_FIREBASE_AUTH_DOMAIN`
- `VITE_FIREBASE_PROJECT_ID`
- `VITE_FIREBASE_APP_ID`

> ⚠️ **deploy.yml 沒注入 `VITE_FIREBASE_RECAPTCHA_SITE_KEY` 與 `VITE_TPEX_PROXY_URL`**。意思是：即使你在 Repo Secrets 設了這兩個，build 時也拿不到。
> - 後果一：**App Check 在 production 從來沒啟用** → Firebase API key 暴露在前端，沒有 reCAPTCHA 保護
> - 後果二：自訂 TPEX proxy 設了等於沒設

### 隱藏陷阱
如果忘了設其他 secrets，build 會「**安靜成功**」（不報錯）但部署後 App 會悄悄降級——沒有 Firebase（純離線模式）、沒有 Finnhub（美股查價爆掉）。所以每次新環境部署完一定要實際打開測。

**替代選擇**：
- **Vercel**：preview deploy（每個 PR 自動產 preview URL）、開發體驗最好
- **Netlify**、**Cloudflare Pages**
- **AWS S3 + CloudFront**：流量大時最便宜
- **自架 Nginx**

## 9. 名詞速查表

| 名詞 | 白話解釋 |
|------|----------|
| **API** | 兩個程式之間溝通的「合約」。例如「我給你股票代號，你回我價格」 |
| **PWA** | 可以裝到手機桌面的網頁 App |
| **SPA** | Single Page Application，整個 App 只有一個 HTML，切換頁面靠 JS |
| **CORS** | 瀏覽器的安全機制：A 網站不能隨便用 JS 去打 B 網站的 API，要 B 網站同意 |
| **Bundle** | 把所有 JS/CSS 打包成幾個檔案，瀏覽器一次下載 |
| **Service Worker** | 瀏覽器背景跑的小程式，負責快取與離線運作 |
| **localStorage** | 瀏覽器內建的小型儲存（5–10 MB），key-value 格式 |
| **IndexedDB** | 瀏覽器內建的「真資料庫」（GB 級、有索引）— 這個 App **沒有用** |
| **Local-first** | 先寫本地、再同步雲端的設計哲學（這個 App **不是真 local-first**）|
| **Last-Write-Wins (LWW)** | 衝突解法：後寫的蓋掉先寫的 |
| **CRDT** | 進階的多人協作衝突解法（後寫的不會無腦蓋掉） |
| **Firestore** | Google 的雲端 NoSQL 資料庫 |
| **realtime listener** | Firestore 的「資料一變就推給所有訂閱者」機制 |
| **Firebase Auth** | Google 提供的登入服務 |
| **App Check** | Firebase 防止 API 被盜用的機制（搭配 reCAPTCHA） |
| **Fallback** | 「備援」：第一個失敗就試第二個 |
| **HMR** | Hot Module Replacement，改 code 不用整頁 reload |
| **CI/CD** | 自動化的 build + deploy 流水線（這個 App 用 GitHub Actions） |
| **outbox** | 「待寄出箱」：先存在本地待網路恢復後再送雲端的 queue（這個 App **目前是 dead code**）|

| 想要即時股價（不只手動更新） | 目前是手動觸發 | 付費 API 或接 WebSocket |
| 想上 App Store | 目前是 PWA | 包 Capacitor 或改寫 React Native |
| 想要更安全（API key 保護） | App Check **沒啟用**（deploy.yml 漏）| **改 1 行 yaml 即可**（高 CP 值）|
| 想要更多自動化（每月帳單匯入） | 沒有 backend | 加 Cloud Functions 或自架 server |
| 想擴展到多人 App | TPEX 走 hardcoded 第三方 proxy | 自架 proxy（Cloudflare Worker）|

每個決策都不是「對 / 錯」，而是「**現在的痛**值不值得付那個**重構成本**」。
